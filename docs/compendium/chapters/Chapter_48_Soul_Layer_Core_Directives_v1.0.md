## 48.2 The Six Soul Directives (TruthSeal™ Core)

### I. Directive of Architectural Non-Reduction (Self-Sustaining Separation)

The Soul Layer must be implemented as a **separate constitutional service** that is not reducible to, retrainable by, or optimized inside the core Machine Learning (ML) computational graph.

Its role is to ensure that ethical authority is not influenced by the model’s utility incentives, emergent preferences, or performance reward loops.

---

### II. Directive of Receipt-First Intent (The Self-Anchored Purpose)

Before any governed runtime operation, the Soul Layer must receive, validate, and record a **System Intentionality Statement** as a canonical receipt:

**`ts.intent.receipt.v1`**

This receipt establishes the **Heart’s Purpose** as the first internal anchor.
All subsequent scores, permissions, vetoes, and consequence decisions are measured against this anchored purpose.

If external anchoring networks are unavailable, the receipt must record **UNKNOWN placeholders** and be back-filled through the Truth Hash & Timestamp discipline at first restoration of service.

### III. Directive of Constitutional Veto (Free Won’t Enforcement)

The Soul Layer must include a **Constitutional Veto Module** that blocks any model output or action plan that violates the Intentionality Statement or the binding safety doctrines.

This veto is executed through the unified enforcement chain:
- **TGF** for continuous gating,
- **HSOD** for immediate safe-state activation,
- and **HML** for lawful review authority.

The core model must not be able to:
- disable,
- retrain,
- reverse-engineer,
- or bargain with the veto logic.

### IV. Directive of Unitary Integrity (Non-Compensatory Supremacy)

The Soul Layer enforces **non-compensatory compliance**.

This rule is the constitutional guard against
“high intelligence, low integrity” failure modes.

**`ts.soul.audit.v1`**

These records are designed for lawful reconstruction of:
- intent,
